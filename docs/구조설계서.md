# c 프 팀플 구조설계서

## 구조 설계서 v 4

## Entity 중심 CLI 미로 탐험 게임 아키텍처

## 1. 설계 목표

본 프로젝트는 C 언어와 Windows CLI 환경에서 동작하는 탐험형 미로 게임을 구현하는 것
을 목표로 한다.
게임은 이모지 기반 렌더링, 상호작용 시스템, 근접 공격, 발사체, 폭탄, 상자, 문, 장애물, 적,
보스 등의 다양한 객체를 포함한다.
이 구조 설계의 핵심 목표는 다음과 같다.
기능 추가 시 기존 구조를 크게 무너뜨리지 않을 것
객체 종류가 늘어나도 공통 인터페이스로 다룰 수 있을 것
렌더링, 입력, 게임 로직, 월드 오브젝트를 분리할 것
상호작용과 전투를 명확히 분리할 것
팀 프로젝트 및 AI 기반 코드 생성 도구가 참고하기 쉬운 구조일 것

## 3. 전체 구조 개요

#### 전체 아키텍처는 다음 흐름을 따른다.

```
Input → Game
→ Map
→ Entity Manager
→ Systems→ Interaction
→ Combat→ Projectile
→ Render→ Log
```
#### 이 구조에서 핵심은 다음과 같다.

```
Map은 지형 타일만 담당한다.
Entity는 월드 위에 존재하는 객체를 담당한다.
Systems는 객체 간 규칙을 담당한다.
Render는^ 출력만^ 담당한다.
Game은 전체를 조율하는 중심 컨텍스트다.
```
## 4. 핵심 설계 원칙


### 4. 1 타일과 엔티티의 분리

#### 맵의 기본 지형은 타일 배열로 표현하고, 상호작용 가능한 월드 오브젝트는 엔티티로 관리한

#### 다.

#### 예:

#### 타일: 바닥, 일반 벽, 부술 수 있는 벽, 출구, 함정 바닥

#### 엔티티: 적, 보스, 상자, 문, 설치된 폭탄, 장애물, 발사체

#### 이 분리를 통해 배경과 객체의 책임이 섞이지 않도록 한다.

### 4. 2 Entity 중심 구조

#### 상자, 폭탄, 문, 장애물, 적, 보스는 모두 월드에 존재하는 객체이며, 위치, 렌더링, 상태, 상호

#### 작용, 피격 여부 등 공통 성질을 가진다.

따라서 공통 Entity 구조체를 두고, 타입별 세부 데이터와 동작을 분리한다.

### 4. 3 시스템 중심 처리

#### 상호작용, 공격, 발사체 처리 등은 특정 객체 파일 내부에 과도하게 몰아넣지 않고, 별도의

```
systems/ 계층에서 처리한다.
```
### 4. 4 출력과 로직의 분리

#### 렌더러는 화면에 무엇을 어떻게 그릴지만 알고, 게임 규칙이나 판정은 알지 않도록 한다.

## 12. 이 구조의 장점

### 12. 1 객체 종류 증가에 강함

#### 상자, 문, 폭탄, 적, 보스, 장애물 등이 모두 공통 루프에서 관리된다.

### 12. 2 책임 분리가 명확함

```
지형은 map
오브젝트는 entity
규칙은 systems
출력은 render
```
### 12. 3 리팩토링이 쉬움

#### 새 기능이 특정 계층에 국한되므로 변경 범위가 줄어든다.

### 12. 4 AI 코드 생성 도구와 궁합이 좋음

모듈 책임이 명확하므로 Codex 같은 도구에 파일 단위 작업을 시키기 쉽다.

## 13. 최종 요약

#### 이 구조는 단순한 C 콘솔 게임 코드 묶음이 아니라,

다양한 월드 오브젝트와 규칙을 수용할 수 있는 Entity **중심** CLI **게임 엔진 구조** 다.
핵심 요약은 다음과 같다.
타일과 엔티티를 분리한다.
모든 월드 오브젝트는 공통 Entity 기반으로 관리한다.
엔티티별 세부 속성은 data 구조체로 분리한다.
상호작용, 전투, 발사체는 systems/에서 담당한다.
렌더링은 출력만 담당하고, 로직은 몰라야 한다.
이 구조는 폭탄, 상자, 문, 장애물, 적, 보스까지 자연스럽게 확장 가능하다.
