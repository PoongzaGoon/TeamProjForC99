## 구조 설계서 v 6

## Stage 데이터 분리형 Entity 기반 CLI 미로 탐험 게임

## 1. 설계 목표

본 프로젝트는 C 언어 기반 Windows CLI 환경에서 동작하는 미로 탐험 게임을 구현한다.
게임은 이모지 기반 렌더링과 Entity 중심 구조를 사용하며, 여러 개의 필드(Map)를 묶은
Stage **기반 탐험 구조** 를 따른다.
이번 개정의 핵심 목표는 다음과 같다.
Map과 Stage의 책임을 명확히 분리할 것
필드 데이터는 사람이 직접 2 차원 배열로 그릴 수 있게 유지할 것
스테이지 확장 시 기존 시스템 코드를 크게 수정하지 않도록 할 것
stage1.c,^ stage2.c^ 등은^ 콘텐츠^ 데이터^ 전용^ 파일로^ 유지할^ 것

## 3. 핵심 구조 원칙

### 3. 1 Map 과 Stage 의 분리


Map은 한 장의 필드만 표현한다
Stage는 여러 개의 Map을 포함하고, 현재 필드 위치와 전환을 관리한다
즉:
Map^ =^ 한^ 화면
Stage = 여러 화면을 묶은 월드/스테이지

### 3. 2 Stage 는 Map 을 상속하지 않는다

```
Stage는^ Map의^ 확장이^ 아니라,^ 여러^ Map을^ 포함 (composition)^ 하는^ 상위^ 관리자다.
```
### 3. 3 필드 데이터와 배치 로직의 분리

```
stage1.c, stage2.c는 필드 배열 데이터만 가진다
```
실제로 어떤 필드를 스테이지의 어느 위치에 넣을지는 (^) stage.c가 결정한다
즉:
stage1.c = 콘텐츠 데이터
stage.c = 조립 및 상태 관리

## 13. Tile 정의

현재 단계에서는 문/열쇠를 Tile 기반으로 유지한다.

```
#define TILE_EMPTY 0#define TILE_WALL 1
#define TILE_DOOR_OPEN 2#define TILE_BOMB_ITEM 3
#define TILE_DOOR_LOCKED 4#define TILE_KEY_ITEM 5
```
장기적으로는 문/열쇠도 Entity로 이동 가능하지만, 현재 단계에서는 테스트와 구현 단순성
을 위해 Tile 기반을 허용한다.

## 15. Interaction 시스템과 Stage 의 관계

#### 상호작용 시스템은 현재 필드를 기준으로 앞칸을 조사한다.

### 처리 예시

#### TILE_KEY_ITEM^ →^ 열쇠^ 획득

#### TILE_DOOR_LOCKED → 열쇠 검사 후 열기

#### TILE_DOOR_OPEN → 필드 이동 트리거 가능

Interaction 시스템은 다음을 직접 관리하지 않는다.

```

stages/*.c = 순수 데이터
```

## 20. 최종 요약

#### 이번 개정 구조의 핵심은 다음과 같다.

```
Map은 필드 한 장을 담당한다.
Stage는 여러 Map을 포함하고, 현재 위치와 진행 상태를 관리한다.
```

stages/stage1.c, stage2.c는 필드 배열 데이터만 저장한다.
stage.c가 그 데이터를 읽어 실제 Stage를 구성한다.
게임의 모든 맵 관련 로직은 항상 **현재 필드** (current field) 를 기준으로 동작한다.
즉 이 설계는 단일 맵 게임이 아니라,
**여러 개의 필드를 연결한** Stage **기반 탐험 게임을 위한 구조** 다.
